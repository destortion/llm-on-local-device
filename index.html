Running a 3.8B LLM on a Phone — A Systems Deep Dive

Architecture, memory analysis, bandwidth math, and real benchmark numbers for Phi-3 Mini running CPU-only on Snapdragon 8 Gen 3 via llama.cpp.

After building a real-time vision AI system with MobileNetV2, I wanted to go further: run a full language model — capable of conversation, summarization, reasoning — entirely on a phone, with no internet, no cloud, no GPU. This post is not just a how-to. It covers the system architecture, memory footprint breakdown, compute vs. bandwidth analysis, and the real hardware constraints that determine performance on a mobile SoC.

1.5 How I Actually Measured This

Every number in this post came from a specific command I ran. This section documents the measurement process — not to pad the post, but because the process itself revealed things I didn't expect.

The Benchmark Loop

I used llama-bench, the built-in benchmarking tool in llama.cpp. Key flags: -p for prompt token count (prefill), -n for output tokens (generation), -t for thread count, -r 3 for 3 repetitions to average out thermal variance.

# Exact command for every data point in Section 6
./build/bin/llama-bench \
  -m models/phi3-mini.gguf \
  -p 128,512,1024 \      # vary prompt lengths
  -n 128 \               # 128 output tokens per test
  -t 1,2,4,6,8 \         # sweep all thread counts
  -r 3                   # 3 reps, take median
What I expected vs what the data said I went in assuming more threads = more speed. The sweep said otherwise. At 4 threads: 8.70 t/s. At 8 threads: 6.81 t/s. Eight threads was 22% slower. I ran it twice to confirm.

The explanation: Snapdragon 8 Gen 3 has 4 performance cores (Cortex-X4, 3.3 GHz) and 4 efficiency cores (Cortex-A720, ~2.3 GHz). With 8 threads, the scheduler assigns work to the slow cores. The fast cores finish their matrix blocks and stall waiting on sync. Meanwhile all 8 cores generate heat together, the thermal governor drops clocks. The optimal configuration is 4 threads on performance cores only — a counterintuitive result that only appeared because I actually swept the parameter space.

Deriving the Bandwidth Number

There's no hardware bandwidth counter available in Termux. I derived it: generation speed (t/s) × model size (GB) = bandwidth consumed per second. At 8.7 t/s × 2.23 GB = 19.4 GB/s. Comparing that against LPDDR5X spec (77 GB/s) showed 25% utilization — which immediately raised the question: if there's 75% bandwidth headroom, why can't I run 4× faster? That question is what the roofline analysis, DRAM row buffer section, and NEON SIMD sections exist to answer.

The question that drove the whole technical analysis I tried adding threads to drive more memory requests — got slower. I tried increasing context — same speed. Eventually the load/store pipeline analysis showed the bottleneck isn't DRAM capacity at all: it's the CPU's ability to issue enough memory load instructions to saturate the bus. DRAM can deliver 77 GB/s. The CPU's effective memory access pipeline tops out at ~19–21 GB/s regardless of thread count. That's the real ceiling — and it's a CPU microarchitecture constraint, not a memory controller one.

The Test That Forced the Model Switch

I started with TinyLlama because it benchmarks at 34.96 t/s — 4× faster. I ran three basic tests before anything else.

Why TinyLlama failed — and what it revealed about model selection TinyLlama 1.1B was trained as a text continuation model. It predicts what comes next in a document — it has no concept of "follow this instruction." Without instruction fine-tuning (RLHF / SFT on chat data), it treats "What is 2+2?" as the start of a sentence to continue, not a question to answer. The failure wasn't about size. It was about training objective. Phi-3 Mini was explicitly trained for instruction-following using curated high-quality data. The 4× speed difference is the price of the model actually doing what you ask. This is the single most important design decision in the whole project — and it was only confirmed by testing, not by reading the benchmarks.

2 Hardware & SoC Analysis

The Snapdragon 8 Gen 3 has multiple compute units. We use only one of them:

We are leaving ~99% of the SoC's compute on the table. The CPU-only path is the only practical option in Termux without a custom build environment — but it's worth understanding what's being sacrificed.

Cache Hierarchy & Why It Matters

The Kryo cores have a 3-level cache hierarchy. The Phi-3 Mini model is 2.23 GiB — it fits in none of them:

Cache LevelSizeLatencyModel fits?
L1 (per core)64 KB~4 cycles✗ No (35,800× too small)
L2 (per core)512 KB~12 cycles✗ No (4,475× too small)
L3 (shared)6 MB~40 cycles✗ No (380× too small)
LPDDR5X RAM12 GB~100 ns✓ Yes — only option
Implication: Every token generated requires reading the full 2.23 GiB of model weights from DRAM. There is no cache reuse between tokens. The CPU waits on memory fetches for the majority of each token's compute cycle — this is a memory bandwidth-bound workload, not a compute-bound one.

The bottleneck is the CPU's ability to feed memory requests — not DRAM bandwidth and not floating-point compute. The CPU can issue ~19.4 GB/s of memory reads at 8.7 t/s, but the DRAM is capable of 77 GB/s. The gap is the CPU's load/store pipeline and cache miss handling overhead. Moving to the Adreno GPU (which has wider memory access paths) could potentially 3–5× this throughput.

Q4_K_M is not simply "4-bit weights." It's a mixed-precision K-quantization scheme from llama.cpp:

Total RAM is not just the model file. The KV cache grows with context length and is a significant second cost:

Minimum requirements: This setup needs at least 6 GB of total RAM (to keep 4+ GB free for inference) and a performance-tier SoC. A budget Android with 3 GB RAM will OOM-kill the llama-server process mid-inference. Tested on Samsung Galaxy S24 with 12 GB RAM — ~3.4 GB was free during benchmarking.

Why KV Cache Matters

The KV cache stores the key and value vectors from the attention mechanism for all previous tokens in the conversation. Without it, the model would need to reprocess the entire conversation history on every new token — O(n²) per response. With the KV cache, each new token only processes its own attention step — O(n) amortized. The tradeoff: memory grows linearly with conversation length.

Thread Count vs Generation Speed

More threads hurt generation speed due to thermal throttling and the Snapdragon's heterogeneous core design (4 performance + 4 efficiency cores). Using all 8 threads pulls in the slower efficiency cores and causes load imbalance.

4 threads — perf cores only8.70 t/s ✓ optimal
8 threads — all cores, thermal throttle6.81 t/s
I started at 8 threads. The benchmark moved me to 4. My first run used -t 8 — the intuitive default. It gave 6.81 t/s. I dropped to -t 4 as an experiment, expecting it to be slower. It gave 8.70 t/s. I ran the 8-thread case again to rule out a measurement artifact — same result. The Snapdragon's big.LITTLE scheduler is the cause: the kernel assigns matrix block work to whichever cores are available, including the A720 efficiency cores at 2.3 GHz. Those cores complete their blocks in ~1.43× the time of the X4 cores, causing the fast cores to stall at the synchronization barrier waiting for them to finish. Adding threads added stalls, not throughput.

A Roofline Model Analysis

The roofline model plots arithmetic intensity (FLOP/byte) against attainable performance. It shows definitively whether a workload is memory-bound or compute-bound on a given platform.

ParameterValueDerivation
FLOPs per token (generation)7.64 GFLOP3.82B params × 2 (multiply-accumulate)
Bytes read per token2.394 GBFull 2.23 GiB model weights, zero cache reuse
Arithmetic Intensity (OI)3.19 FLOP/byte7.64 GFLOP ÷ 2.394 GB
Ridge Point (CPU)3.90 FLOP/byte300 GFLOPS ÷ 77 GB/s peak bandwidth
Ridge Point (Adreno GPU)16.9 FLOP/byte1,150 GFLOPS ÷ 68 GB/s
Reading the roofline: OI = 3.19 FLOP/byte sits left of the CPU ridge point (3.90) — meaning the CPU hits its memory bandwidth ceiling before its compute ceiling. The workload is memory-bound on CPU. On the Adreno GPU (ridge = 16.9), the same OI sits far left of the ridge — it would be even more memory-bound, but the GPU's wider memory bus (68 GB/s vs CPU's ~19 GB/s effective) means higher absolute throughput.
What this analysis actually changed in my approach Before running the roofline, I was looking at the code trying to find a way to optimize the NEON kernel — add prefetch hints, try different loop unrolling, pin threads differently. The roofline analysis made that pointless: OI of 3.19 is left of the ridge, so no amount of compute optimization moves the needle. The only lever that matters is reducing bytes read per token — either by using a smaller model, increasing quantization, or adding GPU bandwidth via a different backend. The analysis didn't give me a fix; it told me exactly where not to spend time. That's what made it useful.

C Prefill vs Decode — The Math

Prompt processing (prefill) and token generation (decode) have fundamentally different compute complexity, which is why their t/s numbers differ by ~3×.

// Prefill: all L prompt tokens computed in parallel
Prefill_FLOP = L² × d   // attention is O(L²·d) — each token attends to all L tokens

// Decode: one new token per step, attends to all prior tokens via KV cache
Decode_FLOP  = L × d    // attention is O(L·d) — KV cache eliminates recomputation

// KV cache memory for Phi-3 Mini
KV_MB = ctx_len × 0.375 MB  // 32 layers × 2(K,V) × 32 heads × 96 dim × 2 bytes
// Example: 512 tokens → 192 MB | 4096 tokens → 1,536 MB

This explains why prompt processing (PP) is measured in t/s at 22–32 t/s, while decode (TG) is 8.7 t/s. During prefill, the attention matrix is computed for all L×L token pairs in one batched operation — GPU/SIMD utilization is high. During decode, only a single new row of the attention matrix is computed per step — SIMD lanes are mostly idle while waiting for KV cache memory reads.

PhaseComplexityMeasuredBottleneck
Prefill (PP)O(L² · d)22–32 t/sCompute — attention matmul is batched
Decode (TG)O(L · d)8.7 t/sMemory — full weight reload per token
TTFT (5 tokens)—228 msPrefill latency dominates short prompts

E Quantization Error Distribution

Q4_K_M applies different precision to different layers based on their sensitivity to quantization error. Layers with high RMS error or high downstream impact get 6-bit treatment; the rest get 4-bit.

The "M" (medium) variant of Q4_K applies 6-bit to ~30% of weights (attention + LM head) and 4-bit to the remaining ~70% (FFN bulk). Total size impact: ~2% larger than pure Q4, ~2% better perplexity than pure Q4.

I DRAM Row Buffer Locality

LPDDR5X DRAM organizes memory into rows (~2KB each). Accessing the same row twice uses a fast "row hit" path. Accessing a different row requires a slow row precharge + activate cycle. The random access pattern of weight matrix reads produces very few row hits.

ParameterValueNotes
DRAM row buffer size2 KBLPDDR5X standard row size
Row buffer hit rate (inference)~15%Weight access pattern is near-sequential within layers but jumps between layers
Row hit latency~15 nsServe from open row
Row miss latency~45 nsPrecharge + activate + read
Effective DRAM bandwidth~41 GB/s0.15 × 77 + 0.85 × (77 × 0.45) = 41.0 GB/s
CPU achieves19.4 GB/sMeasured
Utilization of effective BW47%19.4 / 41.0 — not the DRAM, but the CPU's load/store bandwidth
Reframing the bottleneck: The theoretical DRAM peak is 77 GB/s, but due to row buffer miss penalties, the effective bandwidth available to this workload is ~41 GB/s. The CPU is using 19.4 GB/s of that — 47% utilization. The remaining headroom (~21 GB/s) is not accessible via the CPU's load/store pipeline width. This is why the bottleneck is the CPU's memory access pipeline, not raw DRAM capacity.

J Attention Sinks & Context Limits

Phi-3 Mini, like other modern transformers, exhibits the attention sink phenomenon described in the StreamingLLM paper (Xiao et al., 2023). Initial tokens receive disproportionately high attention weight regardless of content.

ParameterValueNotes
Declared context window4,096 tokensPhi-3-mini-4k-instruct configuration
Attention sink tokens4 tokensFirst 4 tokens always preserved in KV cache
Usable context4,092 tokens4,096 − 4 sink tokens
KV cache at full context1,536 MB4,096 × 0.375 MB per token
StreamingLLM overhead+15–20% latencyDropping middle tokens requires recomputing positional encodings for retained tokens
Why attention sinks matter for mobile: To extend context beyond 4,096 tokens without blowing the KV cache, you'd use a sliding window that evicts middle tokens while preserving the 4 sink tokens. But StreamingLLM's recompute step adds 15–20% latency per eviction cycle — which at 8.7 t/s on CPU is a meaningful pause. The practical limit for smooth conversation on this setup is ~2,048 tokens (~5–10 back-and-forth exchanges) before performance degrades noticeably.

L Engineering Trade-offs & Design Rationale

Every component choice in this system was a deliberate trade-off. This section documents the reasoning and the alternatives that were explicitly rejected.

Why These Components?

Phi-3 Mini 3.8B — not TinyLlama, not Llama-3 8B
Smallest model with reliable instruction-following on Snapdragon hardware. TinyLlama failed basic QA. Llama-3 8B requires ~4.8 GiB (Q4_K_M) — pushes RAM usage to ~5.5 GB, marginal on 8 GB devices and thermally demanding. Phi-3 Mini hits the sweet spot: 2.23 GiB, MMLU 68.8%, designed explicitly for edge deployment.
Rejected: TinyLlama (unusable), Llama-3 8B (OOM risk)
llama.cpp — not Ollama, not MLC-LLM, not ONNX Runtime
llama.cpp compiles natively in Termux with no NDK, no Docker, no root. It provides a self-contained OpenAI-compatible HTTP server. Ollama requires systemd and Docker. MLC-LLM requires a custom build pipeline per model. ONNX Runtime requires model export + runtime install. llama.cpp is the only option that works from a standard Termux shell in under 30 minutes.
Rejected: Ollama (systemd), MLC-LLM (build complexity)
Q4_K_M — not Q8_0, not Q5_K_M, not Q3_K_S
Q8_0 doubles model size to ~4.5 GiB — fits in 12 GB device but OOMs on 8 GB. Speed gain is negligible (<5%) since the bottleneck is bandwidth, not precision. Q3_K_S would shrink to ~1.7 GiB but causes 3–5× higher perplexity degradation and visible coherence drops in long responses. Q4_K_M gives the best accuracy-per-byte ratio in llama.cpp's quant lineup for this model size.
Rejected: Q8_0 (OOM), Q3_K_S (quality drop)
Flask + SSE — not FastAPI, not raw WebSocket, not polling
Flask requires zero async setup, which matters when the bottleneck is the LLM — not the server. SSE is native HTTP (no upgrade handshake) and works on any browser without a JS library. Polling at 100ms intervals would add 100ms jitter to perceived streaming. FastAPI's async advantages only materialise under concurrent load — this setup is single-user by design.
Rejected: FastAPI (overkill), WebSocket (no benefit for unidirectional stream)

Limitations — Real-World Constraints

⚠Accuracy ceiling at ~68% MMLU — Phi-3 Mini, even at FP16, is not GPT-4 class. Complex reasoning chains, multi-step math beyond grade school, and nuanced factual recall degrade noticeably compared to larger cloud models. Expect ~15–20% lower accuracy on hard tasks.
⚠No parallel request handling — llama-server processes one request at a time. A second user (or a second browser tab) queues and waits. Concurrent use requires multiple llama-server instances, doubling RAM usage.
⚠Context window is a hard 4,096 tokens — Phi-3-mini-4k-instruct was fine-tuned with 4K context. Feeding longer conversations silently truncates or causes incoherent outputs. No sliding window or RAG is implemented here.
⚠No streaming cancellation on server side — the Stop button uses AbortController to cut the browser's SSE connection, but llama-server continues generating tokens until completion. The wasted inference is thrown away — it does not free CPU immediately.
⚠OCR is not a substitute for vision — Tesseract fails on handwriting, diagrams, charts, and dense tables. It is a preprocessing step for printed text only. True visual understanding requires a vision-language model (LLaVA, Phi-3 Vision), which is not yet feasible at this RAM budget.
⚠Reproducibility across devices is not guaranteed — thermal throttling, background app state, OS version, and LPDDR variant all affect measured t/s by ±15–20%. The numbers in this post are specific to Galaxy S24, Snapdragon 8 Gen 3, 12 GB RAM, Android 14.

M What I Got Wrong — Corrected Assumptions

The analysis in this post is the result of investigation, not the starting point. Here are the assumptions I went in with, the data that contradicted them, and what I concluded instead. These corrections are where the actual understanding lives.

✗ Initial assumption
The bottleneck is DRAM bandwidth. The model reads 2.23 GB per token and LPDDR5X peaks at 77 GB/s — so maximum speed is 77 ÷ 2.23 = ~34 t/s.
How I got here: basic bandwidth math before running anything.
✓ What the data showed
Measured 8.7 t/s — less than a third of the predicted ceiling. DRAM row buffer analysis showed effective BW is ~41 GB/s (not 77), and the CPU's load/store pipeline caps out at ~19–21 GB/s regardless. The bottleneck is the CPU memory access pipeline, not DRAM capacity.
✗ Initial assumption
8 threads will be faster than 4 threads. More cores = more parallelism = higher throughput. This is how every multi-threaded workload works.
How I got here: intuition from CPU-parallel workloads like rendering and compression.
✓ What the data showed
4 threads = 8.70 t/s. 8 threads = 6.81 t/s. Eight threads was 22% slower. LLM generation is a memory-bound sequential workload — more threads don't create more bandwidth. They add synchronization overhead and pull in slow efficiency cores.
✗ Initial assumption
Compute utilization is the limiting factor. A 3.8B parameter model doing 7.6 GFLOP per token against a 300 GFLOPS CPU should be ~2.5% utilized — there's room to go faster by optimizing the compute path.
How I got here: compute utilization framing, standard in GPU workload analysis.
✓ What the data showed
2.5% compute utilization is correct — but it means compute is not the bottleneck at all. Optimizing FLOP throughput changes nothing. The roofline OI (3.19 FLOP/byte) sits left of the ridge point (3.90) — the model is memory-bound by definition. Compute optimization is irrelevant until the memory access pattern improves.
✗ Initial assumption
A system prompt telling the model to "be concise" would control response length and verbosity reliably.
How I got here: system prompts work this way on cloud LLMs like GPT-4 and Claude.
✓ What the data showed
Phi-3 Mini intermittently ignores system-role messages, especially for slash commands like /summarize. The fix was merging the instruction into the user turn: {role:"user", content: instruction + "\n\n" + text}. Small instruction-tuned models have weaker system-prompt adherence than RLHF-heavy cloud models.
✗ Initial assumption
Q4_K_M means "4-bit quantization with minor precision loss." The quality degradation would be roughly proportional to the bit reduction — a 50% precision drop would cause ~50% quality loss.
How I got here: treating all quantization schemes as equivalent, linear compression.
✓ What the data showed
MMLU degradation vs FP16 is −0.6%. The K-quants system protects quality-critical layers (attention, LM head) at 6-bit while dropping only the FFN bulk to 4-bit. The mixed-precision design absorbs most of the precision loss in the most error-tolerant locations. Quality is nearly indistinguishable from FP16 in practice.
✗ Initial assumption
The Adreno 750 GPU would straightforwardly give 77 GB/s ÷ 2.23 GB ≈ 34 t/s, a clean 4× improvement over CPU. GPU inference should be the obvious next step.
How I got here: GPU bandwidth spec × model size = throughput projection.
✓ What the analysis showed
The GPU projection accounting for dequantization overhead (Q4 → FP16 before each matmul on Vulkan) and driver overhead drops the realistic estimate to ~18 t/s — only 2× over CPU, not 4×. And llama.cpp's Vulkan backend on Android is not production-stable. The 34 t/s headline number would require INT4-native GPU support, which doesn't exist in the current Vulkan backend.
The pattern across all six corrections: the simple first estimate was always optimistic. Real-world constraints — thermal governors, memory access pipelines, dequant overhead, model training objectives — each took a bite out of the theoretical maximum. The gap between "expected from specs" and "measured from hardware" is where systems engineering actually lives.

9 Limitations

⚠Single client only — llama-server handles one request at a time. A second browser tab will queue behind the first.
⚠Context resets on restart — KV cache lives in process memory. Restarting llama-server wipes the entire conversation history.
⚠OCR accuracy degrades on handwriting, low-res images, and non-English scripts. Tesseract is trained primarily on printed Latin text.
⚠No GPU/NPU path — Using only the CPU leaves ~99% of the SoC's AI compute idle. Adreno GPU inference via Vulkan could 3–5× generation speed.
⚠Minimum 6 GB RAM required — budget devices with 3–4 GB RAM will OOM-kill the process. Requires Android 12+ and a Snapdragon 8xx or equivalent SoC.
⚠Thermal throttling at sustained load — extended inference sessions trigger the thermal governor, reducing clock speeds and dropping generation speed by 15–30%.

11 Try It Yourself

Requires: Android 12+, Snapdragon 8xx (or equivalent), 6 GB+ RAM, Termux from F-Droid.

# 1. Build llama.cpp in Termux
pkg install -y cmake ninja clang git python
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp && cmake -B build && cmake --build build -j4

# 2. Download Phi-3 Mini Q4 (2.3 GB)
curl -L -C - -o models/phi3-mini.gguf \
  "https://huggingface.co/microsoft/Phi-3-mini-4k-instruct-gguf/\
resolve/main/Phi-3-mini-4k-instruct-q4.gguf"

# 3. Start inference server (4 threads = optimal)
./build/bin/llama-server \
  -m models/phi3-mini.gguf -c 4096 -t 4 --port 8080

# 4. Start chat UI
cd ~/llm-chat && python app.py
