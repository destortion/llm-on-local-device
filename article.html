01 — Abstract

Abstract

Summary

This article documents the design, implementation, and empirical evaluation of a fully local large language model inference system running on a consumer Android smartphone. The system runs Phi-3 Mini (3.82B parameters, Q4_K_M quantization, 2.23 GiB on-device footprint) via llama.cpp on a Samsung Galaxy S24 (Snapdragon 8 Gen 3, CPU-only, no GPU or NPU). All benchmark figures are directly measured using llama-bench and reproduced from raw log output. Key findings: prefill throughput ranges from 23.88 to 34.63 tokens/second depending on prompt length; token generation sustains 8.70 t/s at the optimal 4-thread configuration; Time-to-First-Token reaches 21.4 seconds at 512-token prompts. The work demonstrates that capable instruction-following LLM inference is achievable on mid-range consumer hardware without cloud dependency, while documenting the hardware bottlenecks and operational constraints that practitioners need to know before deploying similar systems.

02 — Problem Statement

Problem Statement

Most LLM deployment assumes cloud infrastructure: a GPU server, an API endpoint, and a network connection. That assumption breaks in scenarios where connectivity is unreliable or absent, where data must not leave the device for privacy or compliance reasons, or where recurring API cost is not acceptable.

The question this project addresses is specific: can a capable instruction-following language model run entirely on a consumer smartphone in 2024-2026 hardware, without root access, without a GPU backend, and with a setup reproducible in under 30 minutes?

This is not a theoretical question. The constraints are real:

  • Available RAM: 8–12 GB, of which ~4 GB is consumed by Android at idle
  • Compute: CPU-only on a mobile SoC; GPU inference requires proprietary SDK and custom build pipelines not available in Termux
  • Thermal envelope: sustained inference triggers thermal throttling within 3–5 minutes on typical smartphones
  • Model quality floor: the model must actually follow instructions — smaller, faster models that fail basic QA are not usable regardless of their benchmark throughput
What this is not: This is not a comparison of every available edge LLM. It is a deep evaluation of one specific, reproducible configuration that works today, with measured data and explicit documentation of where it fails.

04 — Design Decisions and Tradeoffs

Design Decisions and Tradeoffs

Every component choice involved explicit rejection of alternatives. The reasoning is documented here because the alternatives are not wrong in general — they are wrong for this specific constraint set.

Model: Phi-3 Mini 3.8B — not TinyLlama 1.1B, not Llama-3 8B
TinyLlama 1.1B was tested first. At 34.96 t/s it is 4x faster. However, it is a text continuation model without instruction fine-tuning — it treats "What is 2+2?" as text to continue, not a question to answer. Three basic QA tests failed before a single useful response was produced. Llama-3 8B at Q4_K_M requires ~4.8 GiB of model RAM plus ~1.3 GiB runtime overhead, totalling ~6.1 GiB. This is marginal on an 8 GB device under Android's baseline memory pressure and causes OOM kills under sustained load. Phi-3 Mini was designed explicitly for edge deployment using high-quality synthetic training data. It hits the minimum viable quality floor at the minimum viable RAM footprint.
Rejected: TinyLlama (fails instruction-following) · Llama-3 8B (OOM risk on 8 GB devices)
Quantization: Q4_K_M — not Q8_0, not Q5_K_M, not Q3_K_S
The system is memory-bandwidth-bound during decode. Q8_0 doubles model size to ~4.5 GiB with less than 5% speed gain — the extra bandwidth cost outweighs the precision benefit. Q3_K_S reduces to ~1.7 GiB but causes a measurable perplexity increase and visible coherence degradation in multi-turn conversations. Q4_K_M at 2.23 GiB is the crossover point: it fits comfortably in available RAM, stays within DRAM bandwidth limits, and maintains MMLU accuracy within 0.6% of the full float16 model.
Rejected: Q8_0 (marginal RAM, no speed gain) · Q3_K_S (quality degradation) · Q5_K_M (size/speed tradeoff not better than Q4_K_M for this model)
Runtime: llama.cpp — not Ollama, not MLC-LLM, not ONNX Runtime
llama.cpp compiles natively in Termux using standard package manager tools (cmake, clang) with no NDK, Docker, or root. It provides a self-contained OpenAI-compatible HTTP server that requires zero Python glue for basic inference. Ollama requires systemd, which Android does not expose. MLC-LLM requires model-specific compilation with a custom TVM build pipeline. ONNX Runtime requires model format conversion and separate runtime installation. llama.cpp is the only option that produces a working inference server from a standard Termux environment in under 30 minutes.
Rejected: Ollama (needs systemd) · MLC-LLM (custom build per model) · ONNX Runtime (format conversion + runtime install)
Thread count: 4 — not 8 (all cores)
At 8 threads (measured): 6.81 t/s. At 4 threads (measured): 8.70 t/s. Eight threads is 22% slower. The Snapdragon 8 Gen 3 has 4 performance cores (Cortex-X4, 3.3 GHz) and 4 efficiency cores (Cortex-A720, ~2.3 GHz). With 8 threads, the OS scheduler assigns matrix blocks to efficiency cores. Performance cores finish their blocks and stall waiting for synchronization. The mixed-frequency barrier, combined with all 8 cores generating heat together, triggers the thermal governor earlier and drops overall throughput. 4 threads on performance cores only is not an obvious choice — it was only confirmed by sweeping the parameter space.
Rejected: 8 threads (measured 22% slower due to big.LITTLE synchronization overhead)
API layer: Flask + SSE — not FastAPI, not WebSocket, not polling
The bottleneck is the LLM, not the server. Flask's synchronous model is entirely adequate for a single-user setup. SSE is native HTTP with no upgrade handshake and works in all browsers without a JavaScript library. Polling at any interval adds perceptible jitter to token display. FastAPI's async benefits only appear under concurrent load — this system is single-user by design. WebSockets add a bidirectional protocol for a fundamentally unidirectional data stream (model → user).
Rejected: FastAPI (async overhead with no benefit), WebSocket (bidirectional complexity for one-way stream), polling (visible jitter)

The pp128 Anomaly

pp128 (29.69 t/s) is slower than both pp64 (34.01 t/s) and pp256 (32.72 t/s). This is non-monotonic and requires explanation. The variance figures clarify it: pp32 has ±2.68, pp64 has ±2.94, pp128 has ±0.49, pp256 has ±2.11.

Short runs (<1 second) catch the CPU at transient boost frequencies before the thermal governor intervenes. pp128 at ±0.49 is the most thermally stable measurement — the CPU has reached a steady operating point. The reported "dip" is not a real computational regression at 128 tokens; it is the first prompt length that runs long enough to average out the boost transient. The pp128 number (29.69 t/s) is the most representative sustained throughput figure in the sweep.

Bandwidth derivation. Generation speed × model size gives effective DRAM bandwidth: 8.70 t/s × 2.23 GB = 19.4 GB/s. The Snapdragon 8 Gen 3 has LPDDR5X rated at 77 GB/s — this system achieves 25% utilization. The gap is not available headroom; it is the CPU's memory access pipeline ceiling. Increasing thread count does not close this gap (tested — slower at 8 threads). The bottleneck is the rate at which the CPU can issue load instructions, not DRAM capacity.
08 — Failure Cases

Failure Cases

The following failure modes were observed during development and testing. They are documented because they affect system design and user expectations.

Trigger: 500+ token system prompt
TTFT exceeds 20 seconds. User sees a blank screen with no feedback. Indistinguishable from a crash.
Root cause: prefill throughput at 512 tokens is 23.88 t/s. No prefill progress event is surfaced by the OpenAI-compatible API endpoint. Mitigation: keep system prompts under 64 tokens; add a "processing..." indicator before any request.
Trigger: Sustained inference session >5 minutes
Generation speed drops from ~8.7 t/s to ~6–7 t/s. Device surface becomes warm. In severe cases (direct sunlight, ambient >35 C), speed drops to 5–6 t/s.
Root cause: Snapdragon thermal governor reduces Cortex-X4 clock from 3.3 GHz toward 2.0–2.4 GHz. Estimated degradation: 15–30% throughput reduction. No user-visible warning. Mitigation: monitor response times; consider 2-minute inference pause windows for long sessions.
Trigger: Multi-turn conversation exceeding 3,500 tokens of accumulated context
llama-server returns a context overflow error or silently truncates early messages. Responses begin losing awareness of earlier conversation turns.
Root cause: context window set to 4096 tokens. System prompt + conversation history + response exceeds this. The model does not signal when context is full. Mitigation: implement server-side context length tracking and either summarize old turns or warn the user.
Trigger: Concurrent request from second browser tab
Second request queues behind first. No timeout, no error. The second tab blocks silently until the first completes.
Root cause: llama-server's single-request queue. Flask has no request arbitration. This is a design property of single-user local inference, not a bug. Mitigation: document the single-client limitation; add a request-in-progress UI state.
Trigger: OOM kill during inference
llama-server process is killed by Android's low-memory killer (lmkd). All KV cache is lost. User receives a connection refused error on the next request.
Root cause: model process (~3 GiB) competes with Android system processes under memory pressure. More likely when background apps consume RAM. Mitigation: close background apps before long sessions; 12 GB RAM devices have adequate headroom; 8 GB devices are marginal under typical Android memory pressure.
Trigger: OCR on handwritten text, low-contrast images, or non-Latin scripts
Tesseract returns empty output or heavily corrupted character sequences. The LLM then attempts to answer questions about garbage input, producing plausible-sounding but incorrect responses.
Root cause: Tesseract is trained primarily on printed Latin fonts. Handwriting and non-Latin scripts are outside its reliable operating envelope. No confidence score is surfaced to the user. Mitigation: display raw Tesseract output before sending to LLM; add a minimum character-count threshold before triggering LLM analysis.
09 — Real-World Constraints

Real-World Constraints

Hardware Constraints

ConstraintDetailImpact
DRAM bandwidth ceiling CPU sustains ~19.4 GB/s; LPDDR5X rated at 77 GB/s. Gap is CPU load/store pipeline saturation, not DRAM. Decode speed cannot exceed ~8–9 t/s regardless of thread count or tuning
No cache residency Phi-3 Mini at 2.23 GiB fits in neither L1 (64 KB), L2 (512 KB), nor L3 (6 MB) cache. Every decode pass is a cold DRAM read. Latency per token is DRAM-latency-bound, not compute-bound
big.LITTLE asymmetry 4 performance cores (Cortex-X4) + 4 efficiency cores (Cortex-A720). Efficiency cores run at ~70% of performance core frequency. Using all 8 threads creates synchronization barriers and mixed-speed work partitioning that reduces throughput 22%
Thermal throttling Sustained CPU load triggers thermal governor within 3–5 minutes. Clock reduction from 3.3 GHz toward 2.0–2.4 GHz. 15–30% throughput reduction after sustained sessions. Ambient temperature and charging state affect onset time.
Unused compute Adreno 750 GPU: ~4.7 TFLOPS. Hexagon NPU: ~46 TOPS (INT8). Both unused. CPU-only inference uses approximately 1% of the SoC's AI compute capacity

Software Constraints

  • Android memory management: The low-memory killer can terminate llama-server at any point. No graceful shutdown hook is registered in this configuration.
  • No persistent KV cache: Restarting the llama-server process wipes the entire conversation KV cache. Context cannot be resumed across restarts without serialization support.
  • Single-request queue: llama-server in this configuration handles one inference request at a time. Not a limitation for single-user deployment but rules out multi-user or multi-tab scenarios.
  • Context window fixed at launch: The -c 4096 flag sets context window size at server startup. Changing it requires a restart and full KV cache reset.

Battery Impact

Sustained CPU inference at 4 threads on performance cores is a significant battery draw. estimated based on comparable workloads: continuous inference for 60 minutes draws approximately 15–25% battery on a Galaxy S24 (5000 mAh). Generation-speed-limited workloads (writing long documents) will drain faster than TTFT-limited workloads (short Q&A). Running on AC power is recommended for sessions longer than 15 minutes.

When Local Is the Right Choice

  • Medical, legal, or financial data that cannot leave the device for compliance reasons
  • Offline or intermittently connected environments (field work, travel, restricted networks)
  • High-volume, cost-sensitive workloads where per-token API costs accumulate
  • Applications where user trust requires verifiably zero data transmission
  • Development and experimentation where API rate limits are a constraint

When Cloud Remains Necessary

  • Tasks requiring context windows >4K tokens (document-level analysis, long codebases)
  • Quality-critical applications where 68.8% vs 88%+ MMLU accuracy matters
  • Real-time applications where 21-second TTFT is unacceptable for long prompts
  • Multi-modal tasks (direct image understanding, audio) not supported by this stack

11.5 — Industry Context

Industry Context: Why This Matters Now

This project sits at the intersection of three converging trends in applied AI:

Smaller, more capable models. Phi-3 Mini (3.8B parameters) achieves MMLU scores previously requiring 70B+ parameter models in 2022. Microsoft's training approach — high-quality synthetic data over raw scale — demonstrates that model quality does not scale linearly with parameter count. Phi-3, Gemma 2 2B, and Llama 3.2 3B are all competitive examples of this shift. The implication for edge deployment is direct: capable inference will continue to become available at smaller footprints.

Quantization as a first-class engineering tool. Post-training quantization from FP16 to Q4_K_M reduces model size by 70% with less than 1% accuracy degradation. The K-quant family (K_S, K_M, K_L) in llama.cpp applies different quantization intensities to different weight matrices based on their sensitivity — attention weights are quantized more carefully than FFN weights. This is not a research technique; it is a production deployment primitive that has been validated at scale.

Edge AI infrastructure maturation. llama.cpp, MLC-LLM, ExecuTorch (Meta), and MediaPipe LLM Inference (Google) all reached production-grade stability in 2024. The gap between GPU-server inference and mobile-CPU inference continues to narrow through hardware improvements (LPDDR5X bandwidth, larger CPU caches, increasingly accessible NPU toolchains) and software improvements (speculative decoding, continuous batching, prefix caching).

The practical significance: the system described in this article is not a demonstration or a proof of concept. It is a working local AI assistant running on hardware that is already in consumer hands, built entirely from open-source components, in under 30 minutes. That was not true 18 months ago.

13 — Conclusion

Conclusion

This project demonstrates that instruction-following LLM inference is viable on a consumer Android smartphone today, without cloud infrastructure, GPU backends, or root access. The measured results — 8.70 tokens/second generation, sub-2-second TTFT for short prompts, 2.23 GiB RAM footprint — establish a concrete performance baseline for CPU-only mobile inference on current-generation Snapdragon hardware.

Several findings emerged from measurement that were not predictable from theory:

  • Eight threads is 22% slower than four due to big.LITTLE synchronization overhead — empirically determined, not obvious from the architecture documentation
  • The effective DRAM bandwidth ceiling (~19.4 GB/s) is determined by the CPU's load/store pipeline, not DRAM capacity — there is no thread-count or frequency tuning that closes this gap
  • The pp128 throughput dip in the prefill sweep is a measurement artifact (thermal boost transient) and not a real computational regression at 128 tokens
  • Model selection is binary on quality grounds: TinyLlama 1.1B fails basic instruction-following regardless of its 4x speed advantage

The system's primary operational constraint is TTFT at long prompt lengths: a 512-token prompt requires 21 seconds of prefill. This rules out document-level analysis without prompt chunking and makes long system prompts expensive. All other identified limitations are addressable through software changes within the existing hardware constraints.

The path to meaningful performance improvement requires leaving the CPU: GPU inference via Vulkan could deliver 3–5x decode speed improvement; NPU inference via Qualcomm QNN could reach 10x on quantized operations. Both paths remain accessible from Termux but require custom build environments not documented here.

The key result: a fully capable local AI assistant — one that answers questions correctly, streams responses, handles image OCR, and operates entirely offline — can run on hardware already in consumer pockets. The performance is not competitive with cloud APIs for long-context tasks. For short, privacy-sensitive, or offline workloads, it is entirely sufficient.
