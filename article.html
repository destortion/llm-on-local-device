00 — Context

What Is TTFT and Why Should You Measure It?

When you send a message to a local LLM, two things happen: the model reads your prompt (prefill / prompt processing), then it writes the reply (token generation). These are governed by completely different hardware bottlenecks and have very different feels to the user.

Time to First Token (TTFT) is how long you stare at a blank screen before the first word appears. It's entirely determined by prefill speed and prompt length. In a local-first mobile setup, TTFT is often the dominant friction — generation starts streaming immediately after, so slow generation is more tolerable than a 20-second blank stare.

Prefill
Prompt Processing
Reads all input tokens in parallel. Limited by compute — specifically the attention and FFN flops per layer. Fast but scales with context length.
Decode
Token Generation
Generates one token per forward pass. Limited by memory bandwidth — must reload all 2.23 GiB of weights for each token. Slow but constant rate.

I already had generation speed numbers from earlier. This post is specifically about the prefill side — measuring how TTFT changes as prompt length grows, from 32 to 512 tokens.

The Benchmark Command

I used llama-bench — the built-in benchmarking tool in llama.cpp. It runs multiple warm repetitions and reports mean ± std deviation. For the sweep:

# Prefill sweep: pp32, pp64, pp128, pp256
./build/bin/llama-bench \
  -m models/phi3-mini.gguf \
  -p 32,64,128,256 \      # prompt token lengths
  -n 0 \                  # no token generation — pure prefill
  -t 4 \                  # 4 threads (perf cores only)
  -r 3 \                  # 3 repetitions per config
  -o md                   # markdown output

# Separate pp512 run
./build/bin/llama-bench \
  -m models/phi3-mini.gguf \
  -p 512 -n 0 -t 4 -r 3 -o md

4 threads is the empirically established optimal for this SoC — using all 8 threads pulls in the efficiency cores and causes load imbalance that actually slows things down.

03 — Analysis

Prefill Throughput by Prompt Length

Prefill throughput in tokens/second — higher is better. The max is 34.63 t/s at pp32.

pp32 (32 tokens)34.63 t/s
pp64 (64 tokens)34.01 t/s
pp128 (128 tokens)29.69 t/s
pp256 (256 tokens)32.72 t/s
pp512 (512 tokens)23.88 t/s
The pp128 anomaly. pp128 at 29.69 t/s is slower than pp64 (34.01) and pp256 (32.72) — a clear non-monotonic dip. The standard deviation tells the story: pp128 has ±0.49 (very consistent) vs ±2.68–2.94 for shorter runs. The shorter runs are catching the CPU at higher boost clocks; pp128 is seeing a more "average" thermal state. This is a measurement artifact, not a real computational regression at 128 tokens.

Setting aside the pp128 anomaly, the genuine trend is: prefill throughput holds near-constant up to ~64 tokens, then progressively falls. At pp512, throughput drops 31% from peak.

04 — User Impact

What This Means: Wall-Clock Wait Time

Throughput (t/s) is the hardware metric. What users experience is TTFT — the actual seconds you wait before the first token appears. TTFT = prompt_tokens ÷ prefill_speed.

-- TTFT calculation
TTFT = prompt_tokens / prefill_speed

-- example: pp512
TTFT = 512 tokens / 23.88 t/s = 21.4 seconds

Here's every measured prompt length translated to wall-clock TTFT:

Prompt length Prefill speed Std dev TTFT (seconds) UX verdict
pp32 ~25 words 34.63 t/s ±2.68 0.92 s Instant ✓
pp64 ~50 words 34.01 t/s ±2.94 1.88 s Instant ✓
pp128 ~100 words 29.69 t/s ±0.49 4.31 s Acceptable
pp256 ~200 words 32.72 t/s ±2.11 7.83 s Noticeable
pp512 ~400 words 23.88 t/s ±0.17 21.4 s Long wait ✗

TTFT scales non-linearly with prompt length — because throughput also drops as prompts grow. Going from pp32 to pp512 is a 16× prompt size increase, but a 23× TTFT increase (0.92s → 21.4s).

TTFT as Wait Time (wider = worse)

pp32 (~25 words)0.92 s

pp128 (~100 words)4.31 s
pp256 (~200 words)7.83 s
pp512 (~400 words)21.4 s
The 8-second rule. Research on web UX consistently finds that users lose focus at around 8–10 seconds of waiting with no feedback. pp256 (7.83 s) sits right at that boundary. Any prompt longer than ~200 words will feel slow on this hardware — a streaming "prefilling..." indicator would be valuable UX mitigation for long prompts.
05 — Mechanism

Why Does Prefill Speed Fall at Long Contexts?

The attention mechanism is O(n²) in prompt length for the attention score matrix. Each additional token attends to all previous tokens — so the amount of work grows quadratically, while the amount of parallelism (batch dimension) grows only linearly.

-- Attention complexity per layer
Attention ops = 4 × n2 × d_head + 2 × n × d_model × d_head

-- For Phi-3 Mini: d_model=3072, n_heads=32, d_head=96
pp32: ~38M attention flops/layer
pp512: ~9.66B attention flops/layer → 254× more work

In practice the throughput degradation from pp32 to pp512 is only ~31% (34.63 → 23.88 t/s) because llama.cpp packs tokens into batches and processes the feed-forward layers (which are O(n × d) — linear) in parallel. The quadratic attention is a fraction of the total FLOP budget at small-to-medium contexts.

Memory bandwidth still dominates at these lengths. Even at pp512, moving the 2.23 GiB of model weights is the primary cost — the attention quadratic overhead doesn't become the bottleneck until context windows of thousands of tokens. The ~31% throughput drop at pp512 is real but modest; the 23× TTFT increase is dominated by simply having 16× more tokens to process.

The Thermal/Boost Effect on Short Runs

The pp32 and pp64 results show higher throughput (34+ t/s) but also much higher variance (±2.68–2.94). Shorter benchmarks complete in under 1 second — they're catching the CPU at a transient boost state before the thermal governor intervenes. The pp128 result at ±0.49 is actually the most reliable number: the CPU has settled to a stable frequency. pp256 at ±2.11 is again catching boost transients across its 3 repetitions.

For production sizing, pp128 (29.69 ± 0.49 t/s) is the most trustworthy prefill figure — it's both at a representatively common prompt length and measured at a thermally stable operating point.

06 — Engineering Implications

What This Means for Building Local LLM Apps

1. Keep system prompts short

System prompts are included in every request and contribute directly to TTFT. A 256-token system prompt adds ~7.8 seconds to every single response. Keep system prompts under 64 tokens for acceptable latency — that's about 50 words or 3–4 short sentences.

2. Show a prefill indicator

At pp512, users wait 21 seconds before seeing anything. Without feedback, this feels like a crash. A simple "processing..." or prefill progress bar transforms the experience. llama-server's /completion endpoint reports timings.prompt_ms — you can estimate prefill progress from prompt length and measured throughput.

3. The real bottleneck threshold is ~200 tokens

Under 200 tokens (~150 words), TTFT stays under 8 seconds — tolerable with a spinner. Beyond 200 tokens, TTFT starts crossing into "noticeably slow" territory. For document analysis or OCR pipelines, chunk or summarize inputs before passing to the LLM.

4. Prompt caching changes everything (future)

llama.cpp has experimental prefix caching — if the system prompt is cached in the KV state from a prior call, its prefill cost is zero on subsequent calls. At 4096-token context, this is a major win. Currently unstable in mobile builds, but worth tracking.

Target operating range for this hardware: prompts of 32–128 tokens (1–4 short paragraphs) give TTFT under 5 seconds and throughput above 29 t/s. This is the sweet spot for interactive chat on a Snapdragon 8 Gen 3 CPU-only setup.
07 — Reproduce

Run This Yourself

All you need is Termux, llama.cpp, and the model. These exact commands produce the log files above:

# Requires: llama.cpp built in Termux
# See index.html for full build instructions

# Prefill sweep (pp32 → pp256)
./build/bin/llama-bench \
  -m ~/models/phi3-mini-q4_k_m.gguf \
  -p 32,64,128,256 \
  -n 0 \
  -t 4 \
  -r 3 \
  -o md | tee ~/ttft_sweep.log

# pp512 separately
./build/bin/llama-bench \
  -m ~/models/phi3-mini-q4_k_m.gguf \
  -p 512 \
  -n 0 \
  -t 4 \
  -r 3 \
  -o md | tee ~/ttft_pp512.log
Note on environment: Run with the screen on and plugged into power. Background processes (Android's doze, GC pauses in other apps) can cause outlier samples. The -r 3 flag averages 3 runs — enough to wash out most one-off spikes but not enough to eliminate thermal boost effects on short runs.
